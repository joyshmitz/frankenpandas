br-frankenpandas-z53ed — Series::is_monotonic_increasing/decreasing typed-Int64 fast path
=========================================================================================

Vein: avoid materializing Vec<Scalar> for column-value scans (the value-side
analog of the fp-index lazy-Int64 vein — see [[lazy-int64-materialization-vein]]).

Problem: Series::is_monotonic_increasing/decreasing (fp-frame/src/lib.rs
~8001/8034) call self.column.values(), which boxes every element into a Scalar,
then windows(2) + semantic_le/semantic_ge per pair (enum match-dispatch).

Fix (bit-identical, additive): when Column::as_i64_slice() returns Some, scan the
native &[i64] with p[0] <= p[1] (increasing) / p[0] >= p[1] (decreasing).
as_i64_slice() returns Some only for dtype == Int64 && validity.all(), so:
  - no missing values exist (the Scalar path's is_missing() bail never fires);
  - semantic_le/semantic_ge over two Int64 scalars is exactly i64 <= / >=;
  - windows(2) over len <= 1 yields no pairs => all() is true, matching the
    len <= 1 guard.
Therefore the typed result is identical to the Scalar path for every all-valid
Int64 series. Float64 is intentionally left to the Scalar path (NaN/validity
semantics are sacred and subtler).

New test series_is_monotonic_int64_typed_fast_path_z53ed asserts:
  - as_i64_slice().is_some() (the fast path is actually exercised)
  - ascending [1,2,2,5], descending [9,5,5,-3], unsorted [1,3,2], constant
    [4,4,4], single [7], empty — correct for both predicates
  - agreement with the Scalar fallback on an equivalent Utf8-ordered series
