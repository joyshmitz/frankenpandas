# br-frankenpandas-uza04.91 - static rank-select design rejection

## Candidate

Rejected at proof/design gate: static rank-select / wavelet witness for exact all-pairs Kendall discordance counting.

The desired query for a pair `(i, j)` is:

```text
for each row r in x_order_i prefix P:
    count rows q in P where rank_j[q] <= rank_j[r]
```

A static rank/select bitvector over target-column rank order can answer contiguous rank intervals, and a static wavelet tree over row order can answer contiguous row intervals. The prefix `P` above is neither: it is an arbitrary dynamic row set induced by another column's permutation. Exact support therefore needs one of:

- dynamic membership state over target ranks, which is the already-tried dynamic Fenwick/bitset family, or
- a pair-specific 2D dominance/wavelet structure over `(position_in_order_i, rank_j)`, which rebuilds `O(n log n)` pair work and does not reduce the current per-pair wall.

No runtime source was edited or retained for this bead.

## Graveyard match

The relevant graveyard primitive is succinct rank/select (§7.1): raw bitvectors plus two-level rank indexes support constant-time rank on static contiguous domains. That primitive fits inverted indexes and sorted sequences, but not the arbitrary changing prefix set generated by another Kendall order without adding dynamic membership or per-pair 2D indexing.

## Verdict

Rejected before source edit. Score is below `2.0` because proof-ready exactness requires either repeating forbidden dynamic rank-state families or building pair-specific dominance indexes with the same asymptotic work as the current path.

Do not retry static rank/select over only row order or only rank order for Kendall. Next route: a fundamentally different offline divide-and-conquer dominance engine that shares recursion and rank summaries across target columns in one left-column pass, with a hard fallback to the current per-pair counter if it cannot prove exact byte-identical output.
