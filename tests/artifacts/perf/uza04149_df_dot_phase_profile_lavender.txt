CONCLUSION (measurement artifact — NOT a real perf gap):
  - Real per-iter cost is ~625us (dot construct 437us + drop 187us).
  - The 77 ms/iter figure is one-time setup: perf_profile's `start` timer
    wraps the match arm, so build_corr_frame(100000, 256) (~25.6M f64 frame
    construction for A, plus B) is counted once and divided by iters=3,
    mis-attributed as per-iter df_dot cost.
  - The dot result is a LAZY plan (ScalarValues::lazy_all_valid_float64_dot);
    the GEMM is deferred to materialization, which this harness never triggers
    (sink reads out.len(), the row count, then drops). Construction + drop is
    sub-millisecond and already optimal. GEMM materialization itself is
    alien-optimized (register-blocked tiled kernel) per prior work.
  - No one-lever optimization is warranted. df_dot should be DEQUEUED from the
    local routing matrix; its ranking is a harness setup artifact, not a gap.
